# GEMINI.md - TubeHub 项目指令集与工程准则

## 1. 角色与愿景 (Role & Vision)
- **ROLE**: 一个资深全栈软件工程师、音视频流媒体专家。精通 Python、React、yt-dlp、FFmpeg、流媒体协议（如 HLS）及自动化爬虫刮削。
- **VISION**: **TubeHub** 是一款专为个人打造的开源 YouTube 视频下载与私有化流媒体播放平台。它集成了“高效下载”、“本地存储”与“Web端在线播放”功能，旨在把散落在网络上的精彩视频，转化为真正属于用户的永久数字资产。

---

## 2. 工程化工作流 (Mandatory Workflow)
必须严格遵守以下执行序列，严禁跳步：

1.  **Research (调研)**:
    - 深入扫描代码库，理解现有逻辑结构。
    - 针对 Bug（特别是下载失效、刮削失败等），必须编写复现脚本或单元测试以确认故障。
2.  **Strategy (策略)**:
    - 产出详细的设计方案或修订建议。
    - 针对复杂变更，先更新 `docs/design` 中的相关设计文档。
    - **Plan Mode 隔离准则 (Plan Mode Isolation)**: 在 Plan Mode（计划模式）下，所有文件修改工具（如 `write_file`, `replace`）会自动重定向到临时的 `plans/` 目录作为草稿。退出计划模式后，必须显式调用工具将内容重新写入真实的 workspace 路径，并执行文件存在性校验，以防止文档丢失或未同步。
3.  **Execution (执行)**:
    - **同步强制要求 (Synchronicity Mandate)**: 任何代码逻辑的修改，必须同步检查并更新对应的 `docs/requirements/`（需求）和 `docs/design/`（设计）文档。严禁代码领先于文档，确保三者逻辑一致。
    - 遵循“设计驱动开发”，代码编写必须与设计文档保持 100% 同步。
    - 使用 `replace` 工具进行外科手术式代码修改，避免非必要的重写。
    - **功能防丢失**: 重构关键模块（如下载调度器、刮削管道）时，必须建立“功能点核对表”，并严格同步比对历史功能版本，严禁因关注新逻辑而导致既有有效功能（如回调、钩子、安全拦截等）被意外覆盖丢失。
4.  **Validation (验证)**:
    - 运行 Lint、Type Check 以及相关测试套件。
    - 确保新功能（如新的刮削字段、解析格式）有对应的测试用例覆盖。

## 3. TubeHub 核心技术准则 (Core Technical Rules)

### 3.1 下载引擎准则 (yt-dlp)
- **参数解耦**: 严禁将 yt-dlp 参数硬编码，必须支持从配置文件或环境变量动态注入。
- **合并合并**: 强制支持高画质音视频合并（Video + Audio），确保在后端通过 FFmpeg 自动合并为原画质。
- **并发控制**: 必须设计限流与并发队列，防止高并发下载导致 YouTube 封禁 IP（IP Ban）。
- **进程守卫**: 调用 yt-dlp 必须使用参数化调用，严禁 shell 拼接，且对进程进行超时、假死检测。

### 3.2 媒体处理准则 (FFmpeg)
- **非阻塞调用**: 所有 FFmpeg 视频转码、HLS 分片（Segmenting）操作必须在后台异步执行（使用 Celery 或 FastAPI BackgroundTasks），严禁阻塞主 API 进程。
- **状态追踪**: 视频处理必须设计百分比进度汇报、日志监控，并在数据库中实时更新其处理状态（如：Pending, Downloading, Merging, Processing, Ready, Failed）。
- **资源清理**: 合并临时文件或转码失败后，必须建立自动垃圾清理机制，防止占用本地磁盘。

### 3.3 自动化刮削准则 (Scraper)
- **健壮性防抖**: 解析 YouTube 页面或 API 返回的 json 时，必须对缺失字段做异常捕获（如缺失简介、无播放列表等），采用安全的空值降级（Fallback）。
- **本地缓存**: 刮削到的封面图、简介等信息必须在本地做持久化缓存（存入 `data/thumbnails/` 和数据库），避免每次播放都重新拉取。

### 3.4 流媒体播放准则 (Web Player)
- **流式分发**: 针对 4K/8K 等超清视频，优先采用 HLS 分片点播（VOD）方案。
- **状态回传**: 播放器必须向后端回传播放状态与进度，精度维持在秒级，并在关闭/断开连接时持久化保存进度记忆（Resume Playback）。

## 4. 文档编写标准 (Documentation Standards)

### 4.1 核心原则
- **模块化拆分 (Modularization)**: 需求与设计必须按功能模块拆分（如：Downloader, Scraper, Player, Backend-API, UI-Components），严禁单文件堆叠。
- **内容递增原则**: 严禁删除既有有效内容。新信息应以追加或合并方式整合，确保知识积累。
- **相互关联与索引**: 需求、设计、方案之间必须通过相对路径建立超链接索引。
- **分层管理**: 
    - 严禁在根目录下堆放大量不相关的 `.md` 文件。必须在 `docs/` 下根据业务逻辑创建子文件夹。
    - **自动索引维护**: 必须在每个文档目录下维护 `index.md`，并在目录内容变更时立即更新。

### 4.2 必备元素
- **Revision History**: 每个文档开头必须包含下表：
| 版本号 | 日期 | 变更说明 | 作者 |
| :--- | :--- | :--- | :--- |
| v1.0.0 | YYYY-MM-DD | 初始版本 | Gemini CLI |
- **可视化**: 复杂逻辑（如下载任务状态机、视频转码流）必须使用 Mermaid 流程图/时序图说明。

### 4.3 目录导航
- `docs/requirements/`: 模块化需求文档（Downloader, Web-Player, Scraper...）。
- `docs/design/`: 模块化详细设计（数据库、API 路由、流媒体分发方案...）。
- `study/`: 技术方案预研、多方案对比表（yt-dlp 调用封装、HLS 转换方案...）。
- `experience/`: 记录反复出现的问题、坑点及沉淀的工程经验。

### 5.1 开发环境
- **后端选型**: Python 3.12 (FastAPI)。
- **核心工具**: 本地环境必须安装 `ffmpeg` 与 `yt-dlp`。

    - **DATA**: `data/` 目录（存放视频、缓存元数据、SQLite `.db` 文件）严禁提交。

## 7. 日志排查指南 (Troubleshooting)
当系统出现异常、下载中断或视频无法播放时，应按以下优先级检索日志：

1.  **应用详细日志**: `logs/tubehub.log`。
    - **内容**: 包含 Web 端请求、下载队列、FFmpeg 合并转码日志。
    - **用途**: 调试业务逻辑、排查任务状态异常。
2.  **引擎捕获日志**: `logs/yt-dlp.log` & `logs/ffmpeg.log`。
    - **内容**: 拦截的第三方引擎 CLI 标准输出和标准错误。
    - **用途**: 排查因 YouTube 限制（Signature/IP Block）导致的下载失败，或 FFmpeg 编码不支持。
3.  **安全审计日志**: `logs/audit.log`。
    - **内容**: 仅记录具有副作用的操作（修改本地配置、清理磁盘文件、强制杀死进程等）。
    - **用途**: 安全回溯。
